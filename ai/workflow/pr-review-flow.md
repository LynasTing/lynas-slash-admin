## 自动化预检规则设计

### 代码格式规范

1. 必须符合项目中 eslint or prettier 的格式；
2. 使用 Prettier 格式化代码;
3. 使用 ESLint 监测代码格式;
4. 集合 Husky 进行 commit 预检；

预检执行流程：

1. 本地提交阶段：
   - 使用 Husky + lint-staged 执行 ESLint、Prettier 校验
   - 不通过则阻止 commit

2. CI 阶段：
   - 执行完整 ESLint、TypeScript 检查
   - 执行重复代码检测（SonarQube）
   - 任一失败则阻止 PR 合并

- 拦截策略：
  - 允许自动修复后进行自检，且在检查通过后进行提交;

### 命名规范

1. 常量命名使用全大写，且使用下划线分隔单词；
2. 所有变量、函数名使用小驼峰命名法;
3. 文件夹/文件使用 kebab-case 命名法;
4. 组件使用 PascalCase 大驼峰命名法;
5. 图片使用命名使用全小写，且使用下划线分隔单词;
6. 严禁使用任何的中文或中文拼音命名；

- 拦截策略
  - 阻止提交；

### 注释规范

1. 所有公共方法、Hook必须要有注释；
2. 业务的核心函数、复杂函数的关键实现步骤必须要有注释；
3. 变量使用单行注释；
4. 函数使用多行注释;
5. 不允许在行尾写注释;
6. 函数的参数必须要有注释;
7. 函数的返回值必须要有注释;

- 拦截策略
  - 阻止提交;

### 重复代码监测

1. 使用重复代码检测工具（SonarQube）检测重复逻辑；
2. 当重复率超过阈值（5%）时，CI 阻止合并；
3. 在新增公共函数前先确认是否已有存在的重复函数;
4. 被重复使用代码可根据具体业务逻辑选择是否抽离;

- 拦截策略
  - 阻止提交;

### 无效代码/调试代码监测

1. 禁止提交未使用的变量、函数及其它引用；
2. 禁止提交 console.log、debugger 等调试代码;

- 拦截策略
  - 阻止提交;

### 类型安全监测

1. 如果项目有使用TypeScript，在提交前进行此规则检测;
2. 在提交前进行 tsc 检测;
3. 尽可能的避免使用 any 类型;
4. 如果无法避免使用 any 类型，需要由我确认;

- 拦截策略
  - 阻止提交;
